# Shell / Prime-Branch Interaction Note

> Status: active framing note.
>
> This note applies the branch vocabulary to circle-adjacent shells and primes.
> It does not claim that MoO defines circles or that primes explain geometry.

## Core Reading

Inside the current counting-spine frame, the relevant structures are:

```text
square branch:
  n enters self-relation: n*n

product branch:
  a*b lands on n with a,b > 1

prime branch:
  counting reaches n, and no nontrivial product landing reaches n

shell-relation branch:
  repeated witnessed cases where x, y, and r enter square relation,
  x*x and y*y enter addition, and that result agrees with r*r
```

Composites are product-branch landings.

Primes are prime-branch members. The irreducibility condition is how
prime-branch membership is witnessed; it is not a replacement for the branch.

Circle-like structure is not imported as Euclidean geometry. It begins, if at
all, as repeated shell-relation interaction among square, additive, product,
and prime-branch readings. A circle-like shadow would be cast by that
interaction, not identical to the interaction itself.

## Research Question

The MoO-native question is:

```text
Where do product landings and prime-branch members organize shell-relation cases?
```

Not:

```text
Do primes explain circles?
Does MoO square the circle?
Does x*x + y*y = r*r define a circle?
```

## Why Primes Matter Here

Classically, shell relations and Pythagorean triples are controlled by
factorization. MoO should not import that as an answer. It should use it as a
scrutiny lens:

```text
Do shell-relation cases appear where product structure lands cleanly?
Do prime-branch members constrain which shell-relation cases become visible?
Do primitive Euclid parameters expose a stable relation between product
landings, prime-branch membership, and square/addition agreement?
```

In MoO language, primes are useful because they form a branch where counting
continues while nontrivial product decomposition does not land. That branch may
help explain why some shell-relation cases are witnessed cleanly while others
remain only value-visible, spine-visible, or unwitnessed in the current field
of observation.

## Report Standard

A shell/prime-branch report should translate machine fields back into MoO:

```text
value appears
value appears through counting spine
value enters square branch
value is a product-branch landing
value is a prime-branch member
shell-relation is witnessed
shell/square/product/prime readings interact
circle-like shadow is only proposed after repeated interaction
```

If a report only says that `x*x + y*y = r*r` holds, it is not yet a MoO result.
It is a candidate site for branch interaction.
