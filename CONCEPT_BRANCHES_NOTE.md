# Concept Branches: Squarehood, Trianglehood, and Shape Anchors

> Status: first concept-family study plus anchor clarification and triangle calibration.
>
> This note records MoO-native probes for square and triangle families, then
> separates those families from sharper concept anchors such as "the square" and
> "the triangle." Geometry is not added to runtime semantics.

## Framing

A concept anchor is a compact structural point, while a concept family is the
branch of constructions generated from that anchor or constraint.

This starts from the original failed attempt behind MoO:

```text
Can 2 be proven from 1?
```

The answer is no, if "proven" means carried forward with the same certainty as
`1`. MoO begins from `I think, therefore 1`; `1` is the only certainty. `2`
requires preserving a prior `1` through iteration, which introduces memory and
therefore places `2` at infinite distance from `1`.

That failure is useful because it keeps the geometry language grounded. The
runtime only makes `Ref(1)` primitive; `2` is generated. The philosophical
reading treats this first generated value as the first line-like relation, but
not as a second certainty.

For this framing:

```text
1             -> point / certainty / center
2             -> line / first generated value beyond certainty / infinite distance from 1
the square    -> anchor / fourfold constraint / 4
squares       -> family / squarehood branch / self-product values
the triangle  -> anchor / threefold constraint / 3
triangles     -> family / trianglehood branch / triangular-value relations
infinity      -> unbounded extension / rotation-limit reading
```

So the square report generated by `concept_branch_study.py` is not really a
study of "the square." It is a first study of the **squares family**:

```text
squarehood = values and constructions shaped by self-product
```

The native signatures measured so far are:

- exact rational square value class;
- self-product first witness, `x * x`;
- first round when a self-product route becomes available;
- integer-square odd-increment scaffold where the bounded corpus permits it.

This is not a claim about geometry yet. It is a test of whether MoO can identify
an arithmetic family before adding external interpretations.

The same caution applies to triangles. The triangle report is not a claim that
MoO contains geometric triangles. It asks whether the abstraction

```text
trianglehood = values and constructions shaped by t = r * (r + 1) / 2
```

leaves a measurable native shadow.

The anchor check is intentionally almost trivial: testing "the triangle" means
checking the role of `3`. The family check is separate; it asks whether
triangular-value relations spread beyond that anchor into a measurable cohort.

## Shape Anchors

The likely low-dimensional shape anchors are:

| anchor | concept | likely native role |
| ---: | --- | --- |
| `1` | point / certainty | primitive center, only first-class seed |
| `2` | line / separation | first generated value beyond certainty, iteration begins |
| `3` | triangle / threefold | first closed polygonal constraint, triadic relation, triangular family seed |
| `4` | square / fourfold | square anchor, `C4`/fourfold constraint, area/self-product family seed |
| `infinity` | circle / rotation-limit | unbounded projection, not a runtime value or constructed node |

The "infinity as 2" reading should remain philosophical: `2` is the first move
beyond certainty; in this framing, it is infinite distance from `1`. The corpus
should not treat `2` and infinity as identical runtime objects.

Under the current `|v| <= 4` corpus, `4` is especially important because it is
both:

- a generated integer inside the value bound;
- the largest integer value admitted by the default aperture.

That makes it a natural square/fourfold anchor, but it also means we must be
careful: some of its apparent importance may come from being the boundary of the
chosen corpus.

Likewise, `3` should be treated as the triangle/threefold anchor before any
geometric triangle probe is attempted. The first triangle-family calibration is
recorded below and should not be inferred from the squarehood report.

## Self-Product Timing

For each exact square `s`, the script checks when a rational root `r` is present
and when `r * r = s` first becomes available.

Result:

```text
47 / 49 exact square values appear when self-product first becomes available.
```

The two exceptions are `0` and `1`:

- `1` is the primitive seed;
- `0` appears by `1 - 1` before `0 * 0` can be a construction route.

This is the strongest finding in the first squares-family study. Even when the
saved first witness is not multiplication, the square value usually appears at
the same round that self-product becomes possible.

Example:

```text
1/4 first witness: 1 / 4
self-product route: 1/2 * 1/2
```

`1/4` is not recorded as a self-product first witness, but it appears when the
root `1/2` first makes the self-product route available.

## Integer Square Scaffold

Under the default `|v| <= 4` bound, the only nonnegative integer squares
available as values are:

```text
0, 1, 4
```

The odd-increment scaffold is visible but tiny:

```text
0 + 1 = 1
1 + 3 = 4
```

This is too small to carry the squares family by itself. Larger integer-square
work requires a wider value bound. The rational-square family is therefore the
better first native signal under current bounds.


The integer triangular scaffold is tiny under the current `|v| <= 4` bound:

```text

1

```

The one nontrivial on-time integer scaffold event is `3` itself:

```text
1 + 2 = 3
```

This makes `3` a reasonable threefold anchor, but not yet a strong center.
That statement should be read literally: the anchor test is mostly a test of
`3`, not a test of geometric trianglehood.

Inside the saturated corpus:

```text
anchor 2: first round 1, derivation events 12814
anchor 3: first round 2, derivation events  8811
anchor 4: first round 2, derivation events  6622
```

So `3` is structurally present and more active than `4`, but it does not appear
as a dominant construction center under this simple anchor comparison.

The triangular-value cohort is visible but less clean than the squares family:

```text
triangle cohort median derivation events:        1329
positive non-triangle median derivation events:   676

triangle cohort median multiplication share:        0.2374
positive non-triangle median multiplication share:  0.1718
```

Only `2 / 43` triangular values appear exactly when the current formula-route
projection first becomes available. Many triangular values arrive earlier by
other equivalent arithmetic paths. That matters: trianglehood is accessible as
an abstraction, but it is not currently a clean timing branch in the same way
squarehood is.

The current reading is:

```text
triangle anchor: 3 / first nontrivial additive closure after 1 and 2
triangle family: measurable but diffuse rational triangular-value cohort
native support: derivation prominence and some multiplicative structure
weak point: formula-route timing is not yet a strong signature
```

## Interpretation

Squarehood remains the cleanest concrete example of a concept family in MoO:

```text
concept anchor: 4 / square / fourfold
concept family: squares / squarehood
native support: exact square value + root availability + self-product timing
```

It is not yet a geometric claim. It is an arithmetic map feature.

The useful result is:

```text
MoO can identify a squares family from native construction structure alone.
```

The triangle calibration is still useful, but in a different way:

```text
MoO can recognize a triangular-value cohort, but current data does not make
trianglehood a strong native timing family.
```

That makes squarehood the better foundation for later geometry probes, while
trianglehood is a useful guardrail: not every plausible geometric abstraction
will immediately become a clean MoO branch. The geometric square and geometric
triangle should both be treated as separate anchor/projection questions, not
collapsed into their value families.

## Limits

These studies still use value-class recognition:

```text
p/q is an exact square if p and q are perfect squares
t is triangular if 8t + 1 is an exact rational square
```

That is acceptable for a first concept-family probe, but a stricter future
version should rely more heavily on observed derivation events. The current
native ledgers store first witnesses and operation counts, not every exact
derivation path.

## Next Step

Use squarehood as the first strong map label, and use trianglehood as a
calibration/control label. Next adjacent work:

1. diagonal-like branch: values structurally adjacent to square roots and
   `sqrt2` probes, kept external where irrational;
2. area-like branch: product structure and self-product closure;
3. polygon/circle bridge probes: only after square, triangle, and area branches are
   stable.
